Metrics, explained

Everything the tiles above are computed from, in plain language: what the raw telemetry endpoint is, every metric it currently emits, how to read it yourself, and the service-level objectives this product is held to.

What /metrics is

/metrics is this product's telemetry endpoint in the Prometheus text exposition format — the industry-standard, machine-readable way a service publishes its own runtime numbers. It looks like raw text on purpose: it is a contract for machines, not a page for people. A monitoring system (Prometheus, Grafana Cloud, Datadog, or a plain curl) scrapes it on an interval and turns the counters into rates, graphs and alerts. This dashboard is one such consumer: every tile above is computed live from the same three surfaces — the metric exposition, the operator status snapshot and the readiness probe.

Each metric is one of three types. A counter only ever goes up (total requests served) — consumers derive rates from its deltas. A gauge goes up and down (resident memory right now). A histogram counts observations into cumulative latency buckets, which is how the p95 tile is derived. Series carry labels — the parameters to watch: handler (the route template, never a raw path or UUID, so cardinality stays bounded), method (non-standard tokens are normalised to OTHER), and status (the response class: 2xx/4xx/5xx).

Deliberate exposure: the endpoint is public and unauthenticated by design, like the health probes — a recorded decision (OD-1) for this demo product. It also emits interpreter/process telemetry (Python GC, memory, file descriptors), which is accepted and documented, never a surprise.

How to read it yourself

The shell is the reference read-path — the same one the SLO table in the observability doc names: 

curl -s https://quorum.stackclimb.com/metrics
5xx error rate
Sum every http_requests_total sample, then the subset labelled status="5xx"; the rate is 5xx / total. That is exactly what the error-rate tile does on each refresh.
p95 latency (bucket-derived)
http_request_duration_seconds_bucket counts requests at or under each le ("less-or-equal") bound, cumulatively. The p95 shown above is the bound of the first bucket holding ≥95% of observations — honest but conservative: the true p95 sits at or below it, and the page shows the bound verbatim ("≤ 0.5") rather than implying precision the buckets cannot support.
Counters reset on restart
All _total counters are process-lifetime; a deploy or restart starts them at zero. Consumers (and the rate tile) treat a decrease as a restart, never a negative rate.
