## Suggestions dropped by the factory

- Keep the next deliverable to one user, one workflow, one measurable outcome, and one
  vertical slice.
- Ask only blocking questions first; non-blocking unknowns become assumptions or later
  open questions.
- One skill drives an artifact; other skills review it.
- Security, privacy, compliance, AI safety, and source-of-truth gates beat convenience and speed.
- Do not create implementation code until the problem statement, requirements,
  acceptance criteria, architecture, test strategy, and security checks exist.
- Do not claim Jira/Confluence changes were made unless an authorized Atlassian MCP/API
  tool actually made them.
- External skills are reviewer/advisor inputs only until audited and approved.
- Before creating a new local skill, search existing skills first through `make skill-discover`.
- Register and audit future skills through `configs/external-skill-registry.json` and
  `make skill-onboarding-check`.
- Use `make handoff` before closing a Codex session or switching terminals.
- After each major phase, run `make validate`.
- Before release readiness, run `FACTORY_STRICT=1 make validate-strict`.
- Publish study artifacts only after the MVP and most valued outcome are clear.
- Git is the source for study modules; Confluence mirrors them after explicit confirmation.
- Every public artifact must explain AI value, work-easing, security, scalability, and
  enterprise readiness with evidence.
- Diagrams, GIFs, videos, FAQ, technical articles, and LinkedIn posts must pass
  media/publication review before publishing.
