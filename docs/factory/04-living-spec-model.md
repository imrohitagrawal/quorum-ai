# Living Specification Model

## Purpose

The living spec prevents Codex from building from vague or stale requirements.

## Required artifacts

- `docs/10-functional-requirements.md`
- `docs/11-non-functional-requirements.md`
- `docs/12-acceptance-criteria.md`
- `docs/17-requirement-registry.md`
- `docs/18-requirement-traceability-matrix.md`
- `docs/19-change-control-log.md`
- `docs/19-signoff-record.md`

## Requirement fields

Each requirement must include:

- Requirement ID
- Title
- Source
- Business owner
- User persona
- Priority
- Rationale
- Acceptance criteria
- NFR mapping
- Security/privacy impact
- Test mapping
- Implementation slice
- Jira link or Jira key placeholder
- Confluence link or page placeholder
- Status
- Last changed date
- Open questions

## Rule

No implementation starts for a requirement until it has acceptance criteria, test mapping, and implementation-slice mapping.
