# Confluence Operational Guide Model

## Purpose

Every meaningful feature must have an operational guide page that helps users, support, QA, SRE, and engineering understand how to use, validate, operate, and troubleshoot it.

## Mandatory Jira-before-Confluence rule

Before creating or updating a Confluence operational guide page, the factory must create or update a Jira item that states:

- why the Confluence page is needed;
- which feature or requirement it documents;
- owner and reviewer;
- target audience;
- required sections;
- acceptance criteria for the page;
- link to the Confluence page after creation.

## Required Confluence page sections

A feature operational guide page must include:

1. Page purpose.
2. Linked Jira requirements.
3. Linked requirement IDs.
4. Business problem.
5. User workflow.
6. Step-by-step feature usage.
7. Expected behaviour.
8. Known limitations.
9. Error scenarios and troubleshooting.
10. Security/privacy notes.
11. Observability and support signals.
12. Release/version applicability.
13. FAQ.
14. Educational awareness section.
15. Change history.

## Educational awareness section

Each Confluence guide must include a separate educational section covering:

- technology used;
- design pattern used;
- architecture style;
- build and deployment approach;
- testing approach;
- observability approach;
- AI usage, if applicable;
- risks and safe usage guidance;
- glossary for non-expert readers.
