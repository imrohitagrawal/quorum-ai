Start with your problem

You should not need to choose skills manually. Add your idea, then ask Codex to start the factory.

python scripts/capture_idea.py "Your rough product idea here"
make next

Start product factory from my idea. Ask clarifying questions first. Do not write code yet.

The factory asks blocking questions, builds the problem statement, identifies the MVP and most valued outcome, then routes the next skill.

External skills first

Before inventing a custom skill, the factory checks existing skills from skills.sh, official providers, and reputable GitHub repositories.

make skill-discover
python scripts/audit_external_skill.py path/to/skill-folder
python scripts/onboard_external_skill.py --name skill-name --source-url https://example.com/repo

Multiple tabs and handoff

Use one workstream per terminal. Use git worktrees for parallel sessions. Do not edit the same artifact family in two sessions.

git worktree add ../my-product-architecture -b feature/architecture
git worktree add ../my-product-implementation -b feature/implementation

Before ending a session:

make next
make skill-route
make handoff
git status

Lifecycle

Idea → Clarifying questions → Problem statement → MVP/value outcome → Requirements → Jira/Confluence → Architecture/security/testing → Implementation slices → Release evidence → Study artifacts → FAQ/article/LinkedIn → Operate and improve

FACTORY_STRICT=1 make validate-strictStrict evidence mode before release.

Hard rules

  • External skills are accelerators, not authority.
  • Search existing skills first; audit before install/use.
  • One local driver owns each artifact; reviewers critique.
  • Jira/Confluence writes need explicit confirmation and post-write review.
  • Session context lives in files and commits, not chat memory alone.
  • Contract changes follow additive-only and breaking-change checks.
